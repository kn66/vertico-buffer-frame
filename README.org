#+title: vertico-buffer-frame

Experimental child-frame display for Vertico's ~vertico-buffer-mode~, with an
optional candidate preview child frame.

The package enables ~vertico-buffer-mode~ and replaces
~vertico-buffer-display-action~ with a display action which creates fresh child
frames for each minibuffer session.  The candidate frame is centered in the
parent frame and sized from a golden-ratio rectangle.  When preview is enabled,
preview content is shown in a second child frame overlaid on the lower-right
side of the candidate frame.

Child frames require a display where Emacs can create child frames.  This is
normally a graphical display, or a terminal display with Emacs'
~tty-child-frames~ support.

This package was split out from the same line of experiments as
~vertico-posframe-preview~, but it has a different constraint: keep the
minibuffer input prompt as wide as possible.  File operations such as renaming,
writing, or opening long paths become awkward when the prompt area is narrowed,
so the preview frame is intentionally overlaid on the lower-right part of the
candidate frame instead of being placed side by side with the input area.

* Usage

Add this directory to ~load-path~, then:

#+begin_src elisp
  (require 'vertico-buffer-frame)
  (vertico-buffer-frame-mode 1)
#+end_src

* Options

- ~vertico-buffer-frame-border-width~
- ~vertico-buffer-frame-golden-ratio-scale~
- ~vertico-buffer-frame-preview~
- ~vertico-buffer-frame-preview-delay~
- ~vertico-buffer-frame-preview-width~
- ~vertico-buffer-frame-preview-height~
- ~vertico-buffer-frame-preview-categories~

Additional preview target adapters can be added with the hook variable
~vertico-buffer-frame-preview-target-functions~.  Hook functions receive
~CATEGORY~, ~CANDIDATE~ and ~RAW-CANDIDATE~, and should return a target accepted
by the internal preview renderer, or nil.

The candidate frame is currently sized from the parent frame using the golden
ratio.  ~vertico-buffer-frame-width~ and ~vertico-buffer-frame-height~ are
defined, but the current display path does not use them.

The preview frame is sized automatically from the candidate frame so it follows
parent frame resizing.  ~vertico-buffer-frame-preview-width~ and
~vertico-buffer-frame-preview-height~ can be set to character counts to cap the
automatic preview size.

Example:

#+begin_src elisp
  (setq vertico-buffer-frame-preview t
        vertico-buffer-frame-preview-delay 0.2
        vertico-buffer-frame-preview-width nil
        vertico-buffer-frame-preview-height nil
        vertico-buffer-frame-border-width 1
        vertico-buffer-frame-golden-ratio-scale 1.0)
#+end_src

To toggle the preview while completing, bind the toggle command in
~vertico-map~:

#+begin_src elisp
  (with-eval-after-load 'vertico
    (define-key vertico-map (kbd "C-t")
                #'vertico-buffer-frame-toggle-preview))
#+end_src

* Commands

- ~vertico-buffer-frame-cleanup~
- ~vertico-buffer-frame-toggle-preview~

When called during completion, ~vertico-buffer-frame-toggle-preview~ changes
preview visibility only for the current minibuffer session.  When called
outside the minibuffer, it changes the global default
~vertico-buffer-frame-preview~.

~vertico-buffer-frame-cleanup~ deletes child frames currently owned by active
minibuffer sessions.

* Stability notes

This package intentionally builds on ~vertico-buffer-mode~ and Emacs child
frames, but it still depends on some Vertico internals.  Candidate lookup uses
~vertico--candidate~ and ~vertico--metadata~, and frame reveal is hooked with an
~:after~ method on ~vertico--display-candidates~.  Consult support is isolated
in ~vertico-buffer-frame-consult.el~ and uses Consult text properties plus a
small amount of Consult imenu internals.  These hooks may need adjustment when
Vertico or Consult changes their internal APIs.

* Preview

Built-in preview targets currently cover files, project files, directories,
buffers, bookmarks, xref locations, imenu entries, symbols and documentation-like
categories such as commands, functions, variables, faces, minor modes, Custom
groups and Custom variables.  They also cover Unicode names, environment
variables, Info menu items, calendar months, colors, themes, input methods,
coding systems, charsets, libraries, packages, registers, processes, email,
ecomplete, dabbrev, BibTeX keys and strings, and kill-ring entries.

Consult-specific targets live in ~vertico-buffer-frame-consult.el~ and are
loaded by the preview module.  They add support for ~consult-location~,
~consult-grep~, ~consult-xref~ and Consult imenu candidates.

Preview display is category-gated by
~vertico-buffer-frame-preview-categories~.  Remove categories from that list to
disable their preview, or add a function to
~vertico-buffer-frame-preview-target-functions~ for extra preview target
resolution.

The current prototype creates a fresh preview frame and a fresh temporary
preview buffer for each preview update.  File previews read the selected file
directly, directory previews list the directory directly, and there is currently
no built-in file-size limit, binary-file detector, or remote-file guard.

* Difference from vertico-posframe-preview

[[https://github.com/kn66/vertico-posframe-preview][vertico-posframe-preview]] is a related package which adds a second posframe for
preview content beside the candidate posframe shown by ~vertico-posframe~.  This
package does not depend on ~posframe~ or ~vertico-posframe~; it uses
~vertico-buffer-mode~ and plain Emacs child frames instead.

The main differences are:

- ~vertico-buffer-frame~ depends only on Emacs and Vertico, while
  ~vertico-posframe-preview~ depends on ~posframe~ and ~vertico-posframe~.
- ~vertico-buffer-frame~ keeps the minibuffer prompt width intact and overlays
  the preview on the lower-right side of the candidate frame.  This favors long
  input workflows, such as file rename and path editing.
- ~vertico-posframe-preview~ places candidate and preview frames side by side
  by default and follows the ~vertico-posframe~ configuration model.  This can
  make large previews easier to read, but it is a different layout tradeoff.
- ~vertico-buffer-frame~ avoids advising ~vertico-posframe~ and Consult preview
  state.  It previews candidates from Vertico metadata, text properties, and
  category-specific target functions.  Consult support is handled by a small
  target adapter instead of mirroring Consult preview state directly.

The reason for this package is therefore not to replace
~vertico-posframe-preview~ feature-for-feature.  It is to explore a lighter
child-frame display path for Vertico which preserves the prompt area and avoids
the extra posframe dependency.
